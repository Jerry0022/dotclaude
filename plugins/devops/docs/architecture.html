Architecture Documentation devops v0.18.2

A comprehensive DevOps automation plugin for Claude Code — 10 hooks, 10 skills, 10 agents, 2 MCP servers, a deterministic completion card engine, and a 3-layer extension system.

3-Layer Extension Model

Every skill supports override/extension at three layers. Higher layers take priority:

Layer 3 (HIGHEST) Project overrides {project}/.claude/skills/{name}/SKILL.md + reference.md Layer 2 Global overrides ~/.claude/skills/{name}/SKILL.md + reference.md Layer 1 (LOWEST) Plugin defaults devops/skills/{name}/SKILL.md
How Extension Loading Works (Step 0)

Every skill begins with a silent extension check:

  1. Check ~/.claude/skills/{name}/SKILL.md + reference.md (global)
  2. Check {project}/.claude/skills/{name}/SKILL.md + reference.md (project)
  3. Merge: project > global > plugin defaults

Extensions can add steps, override steps, or provide project-specific context (build commands, version file paths, deploy targets). If no extension files exist, the skill runs with plugin defaults only.

Extension Example: /devops-ship for an Angular Project 
# {project}/.claude/skills/devops-ship/SKILL.md
## Project-Specific Quality Gates
- Run `ng build --prod` before PR
- Run `ng test --watch=false`

# {project}/.claude/skills/devops-ship/reference.md
## Version files
- package.json -> "version" field
- projects/myapp/package.json -> "version" field

## Deploy target
- SSH: scp dist/ user@192.168.178.32:/var/www/

Hooks (10 Lifecycle Triggers)

Hooks are JavaScript files that fire at specific lifecycle events. They are registered in hooks/hooks.json and follow a naming convention: {event-prefix}.{domain}.{action}.js

Exit Code Convention

CodeMeaningEffect
0AllowProceed normally
1ErrorNon-fatal, tool still proceeds
2BlockPrevent tool execution

SessionStart (3 hooks)

SS ss.tokens.scan.js — Project Size Scanner

Scans the project for expensive files (large binaries, node_modules, media) and reports total size. Fires once per session. Helps the user understand token budget impact of their project structure.

TriggerSession start
GuardrunOnce()
OutputSize report (stdout) or silent if clean
SS ss.git.check.js — Stale Changes Detector

Checks git status for uncommitted changes, unpushed commits, and stash entries. Reports a brief warning if the repo is dirty.

TriggerSession start
GuardrunOnce()
Output"3 unpushed commit(s)" or silent if clean
SS ss.flow.selfcalibration.js — Self-Calibration Scheduler

Registers a 30-minute recurring cron task for self-calibration and executes it immediately on first run. The cron task reviews feedback rules, reads deep-knowledge files in rotating batches, and ensures Claude stays aligned with plugin conventions.

TriggerSession start
GuardrunOnce()
CreatesCronCreate instruction (30-min recurring)

PreToolUse (1 hook)

PRE pre.tokens.guard.js — Token Budget Guard

Checks token burn rate against budget before file-reading operations. Can block (exit 2) expensive operations when burn rate exceeds threshold.

TriggerBefore Read, Bash, Glob, Grep
MatcherRead|Bash|Glob|Grep
Exit 0Budget OK, proceed
Exit 2Budget exceeded, block operation

PostToolUse (2 hooks)

POST post.flow.completion.js — Code Edit Tracker

Tracks code edits (Write, Edit, Bash mutations) and accumulates a changes count. This state feeds into the completion card system to determine what was changed during a task.

TriggerAfter every tool use
MatcherAll tools (no filter)
OutputUpdates internal completion flow state
POST post.flow.debug.js — Failure Detector

Detects repeated Bash failures (error patterns). After 2+ consecutive failures, recommends the /devops-flow skill for root-cause analysis.

TriggerAfter Bash tool use
MatcherBash
Threshold2+ consecutive failures
OutputSkill('flow') recommendation

UserPromptSubmit (4 hooks)

PROMPT prompt.git.sync.js — Periodic Git Sync

Runs git pull origin main --rebase every ~15 minutes if the working tree is clean. Skips if on a feature branch or if there are conflicts. Silent on success.

PROMPT prompt.issue.detect.js — Issue Reference Detector

Scans the user's message for GitHub issue references (#NNN). Updates issue status to "In Progress" if work on that issue is detected.

PROMPT prompt.ship.detect.js — Ship Intent Router

Detects shipping keywords ("ship", "ab damit", "fertig", "merge it") and affirmations after a completion card ("ja", "yes", "go"). Routes to the /devops-ship skill via stdout instruction. Enforces that shipping always goes through the skill, never manually.

Direct keywordsship, ab damit, fertig, merge it, das kann rein, release
Affirmationsja, yes, mach, go, klar (after completion card)
OutputSkill('ship') instruction
PROMPT prompt.flow.appstart.js — App Start Guard

Detects app start intent ("run the app", "start the server", "npm start"). Validates that work was completed first (checks for uncommitted changes). Enforces completion card before app start.

Stop (1 hook)

STOP stop.flow.guard.js — Session End Guard

Checks for lingering background agents or incomplete work when the session ends. Issues a carry-over instruction: the completion card must be rendered as the first output in the next turn. Non-blocking — user can force exit.

/devops-ship — Full Shipping Pipeline

Skill 5 MCP Tools v0.2.0

Complete 8-Step Pipeline
  1. Session Guard — Check for running background agents, pending Bash commands, incomplete TodoWrite tasks. Non-blocking: wait / proceed / cancel.
  2. Load Extensions — Merge global + project overrides (Step 0)
  3. Pre-Flight — ship_preflight MCP: clean tree, commits ahead, versions consistent, worktree detection
  4. Build + Quality Gates — ship_build MCP: build, lint, tests, build-ID. Optional Codex review for major bumps.
  5. Version Bump — ship_version_bump MCP: bump version in all tracked files (package.json, plugin.json, marketplace.json, README). Ask user for major versions.
  6. Release — ship_release MCP: commit → push → PR → squash-merge → tag → GitHub release
  7. Cleanup — ship_cleanup MCP: delete branch, prune worktrees. ExitWorktree if applicable.
  8. Completion Card — render_completion_card with variant "shipped"
MCP Tools Used
ToolReturns
ship_preflight{ ready, checks[], errors[] }
ship_build{ success, buildLog, lintLog, testLog, buildId }
ship_version_bump{ success, vOld, vNew, filesUpdated, verified, mismatches }
ship_release{ branch, commit, pushed, pr, merged, tag, release }
ship_cleanup{ success, deleted[], warnings[] }

/devops-commit — Conventional Commits

Skill v0.1.0 disable-model-invocation

8-Step Flow
  1. Load Extensions — Step 0
  2. Detect Intent — Amend vs. normal commit
  3. Survey Changes — git status --short
  4. Smart Staging — ≤5 files: show diffs & stage all. >5 files: ask user to pick groups.
  5. Read Staged Diff — git diff --cached
  6. Determine Type & Scope — Conventional types: feat/fix/refactor/perf/test/docs/chore/style. Scope from file paths.
  7. Write Message — ≤72 chars subject, optional body (wrap 72 chars), explain why
  8. Execute — git commit or git commit --amend with Co-Authored-By trailer

Special: disable-model-invocation: true means this runs as a pure skill without LLM interpretation of its steps.

/devops-flow — Root-Cause Analysis & Diagnostics

Skill v0.1.0 alias: /debug

8-Step Diagnostic Flow
  1. Load Extensions — Framework-specific log paths
  2. Triage — Ask user if context is empty: "Error in console?", "Crash on startup?"
  3. Check Recent Changes — git log --oneline -5, git diff HEAD~1 --stat
  4. Discover & Read Logs — Glob for **/*.log, framework-specific paths
  5. Locate Failure — Grep for error string, follow stack trace
  6. Root Cause Analysis — Find broken invariant (data/logic/config issue)
  7. Decide: Fix or Propose — Trivial → fix immediately. Unclear → 2-3 hypotheses. Architectural → report only.
  8. Report — Root cause (file:line), what was wrong, what was fixed, confidence level

Can be triggered automatically by post.flow.debug hook after 2+ consecutive Bash failures.

/devops-deep-research — Multi-Angle Research

Skill v0.1.0 context: fork

7-Step Research Flow
  1. Load Extensions — Custom research paths/sources
  2. Depth Check — Simple fact → skip. Multi-faceted → proceed. Ambiguous → full research (3 angles).
  3. Budget Awareness — If burn-rate > 1.3, warn user (~50K tokens)
  4. Break Down Topic — Identify 3-5 research angles
  5. Search & Fetch — Per angle: WebSearch, 2-3 results, prefer primary sources
  6. Cross-Reference — Note agreements and conflicts
  7. Adaptive Output — TL;DR, findings by angle, recommendations

Context: fork — spawns an isolated research agent to keep main context clean.

dotclaude-ship (5 Tools)

MCP mcp-server/devops-ship/index.js

Tool: ship_preflight

Purpose: Safety checks before shipping

Checks: Clean tree, commits ahead of remote, all pushed, version consistency across manifests, worktree detection

Returns: { ready: boolean, checks: string[], errors: string[] }

Tool: ship_build

Purpose: Build + lint + test + build-ID

Returns: { success, buildLog, lintLog, testLog, buildId }

Build-ID is a deterministic content hash over source files (excludes config, docs, build artifacts). Same source = same hash.

Tool: ship_version_bump

Purpose: Bump semantic version in all tracked files

Files: package.json, plugin.json, marketplace.json, README badge

Returns: { success, vOld, vNew, filesUpdated, verified, mismatches }

Verifies all files were updated correctly after bump. Reports mismatches if any file still has the old version.

Tool: ship_release

Purpose: Full release pipeline

Steps: commit → push → PR create → squash-merge → tag → GitHub release

Returns: { branch, commit, pushed, pr: {number, url}, merged, tag, release }

Tool: ship_cleanup

Purpose: Post-release cleanup

Actions: Delete feature branch (local + remote), prune worktrees

Returns: { success, deleted[], warnings[] }

Includes explicit remote branch verification before deletion to prevent accidental removal of wrong branches.

Scheduled Tasks

CRON self-calibration — 30-min Recurring

Registered by ss.flow.selfcalibration.js hook at session start. Runs every 30 minutes.

  1. Self-Audit — Review feedback rules, check session behavior. Completion flow is mandatory after every task (no discretionary skips).
  2. Learn — Scan for new feedback patterns, save as feedback_*.md memory files
  3. Proactive Check — Prepare for upcoming actions (completion cards, ship flow, browser tasks)
  4. Skill Internalization — Read deep-knowledge files in rotating batches (25% per cycle, alphabetically)

Token budget: 5K-15K per cycle

SS stale-changes-check — Session Start Trigger

Not a cron task — triggered by the ss.git.check.js SessionStart hook.

Checks: git status --porcelain (uncommitted), git log --branches --not --remotes (unpushed), git stash list (stash entries)

Clean: Silent. Issues found: One-line prompt warning.

Build-ID Computation

scripts/build-id.js computes a deterministic content hash over source files, excluding config, docs, and build artifacts. Same source content always produces the same hash. Used for:

  • Test deduplication (skip tests if build-ID hasn't changed)
  • Completion card title (visual fingerprint)
  • Build logs (traceability)
