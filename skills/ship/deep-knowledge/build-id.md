# Build-ID Specification

## What it is

A **content hash** over source code + assets. Independent of git commits.
Same source = same hash (idempotent, deterministic).

Generated by: `node scripts/build-id.js`

## What it hashes

**Included**: Source code files, assets (images, fonts, templates)
**Excluded**: Config files, docs, lock files, build artifacts, dist/

## When to generate

| Situation | New build-ID? |
|---|---|
| App started (by user or AI) | Yes — display as `Build a3f9b21 gestartet` |
| Feature is testable (user could try it) | Yes |
| Ship flow (Step 2: Quality Gates) | Yes — after successful build |
| Code doesn't compile | No |
| Typo/comment fix, no functional change | No |
| WIP commit, code half-finished | No |

**Rule of thumb**: Could someone start the app and see the difference? Yes = new build-ID.

## Where it appears

| Location | Shown |
|---|---|
| Completion Card title | `## ✅ a3f9b21 · Summary` |
| Console log on app start | `Build a3f9b21 gestartet` |
| BUILDLOG.md entry | `## a3f9b21 — YYYY-MM-DD` |
| package.json `buildId` field | `"buildId": "a3f9b21"` |
| README.md | No (version only) |
| Git tags | No (version only) |

## Build Log (BUILDLOG.md)

Every shipped build gets an entry:

```markdown
## a3f9b21 — YYYY-MM-DD
Version: x.y.z
Branch: feat/42-name
PR: #42
Commit: e8f2a1c
Changes:
- Change 1
- Change 2
```

Newest at top. Created retroactively if missing.
