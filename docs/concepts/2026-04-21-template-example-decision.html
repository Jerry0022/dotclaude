Auth-Middleware Redesign

Entscheidung zwischen drei Authentifizierungsansätzen

Iteration 1 · Variantenvergleich

Drei Middleware-Ansätze für die neue Auth-Schicht. Stimme pro Variante ab — "Exakt diese" wählt genau eine aus und verwirft die anderen.

Variante A · JWT stateless

Token wird client-seitig gehalten, Signatur via Edge-Worker verifiziert. Keine Session-Tabelle.

Pros
  • Horizontal skalierbar
  • Kein DB-Roundtrip
  • Stateless Edge-Ready
Cons
  • Revocation kompliziert
  • Token-Größe
  • Kein serverseitiger Logout

Variante B · Session Cookie (Redis-backed)

Klassisches Session-Cookie mit signed ID, Mapping in Redis.

Pros
  • Einfacher Logout
  • Revocation sofort
  • Klein auf Wire
Cons
  • Redis als SPOF
  • Session-Sticky-Routing nötig

Variante C · OAuth Delegation (Auth0/Clerk)

Delegation an externen Identity Provider, nur Token-Validierung lokal.

Pros
  • Compliance out-of-the-box
  • SSO + MFA gratis
  • Weniger Eigenbetrieb
Cons
  • Vendor-Lock
  • Kostenfaktor pro MAU
  • Externe Latenz
